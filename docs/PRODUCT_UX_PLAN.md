## 2. 产品定位

PlumeLens / 鉴翎 不是通用图库，也不是传统修图软件，而是：

**面向鸟类摄影用户的桌面端智能筛片工作台 + 长期物种沉淀系统。**

它主要解决两个问题：

1. 我这次拍回来的一个文件夹里，哪些照片值得留下、导出、继续精修？
2. 我长期拍到了哪些鸟，每个物种留下了哪些代表作品与记录？

对应的产品结构为：

- `开始`
  首次进入、导入文件夹、继续上次工作、查看系统是否准备就绪
- `选片`
  当前主工作界面，以文件夹为单位进行筛片
- `羽迹`
  长期沉淀模块，承接跨文件夹照片浏览与物种记录

## 3. 两条主体业务线

### 3.1 业务线 A：照片识别筛选

这是当前最高优先级主线。

完整流程为：

1. 用户拍摄一批新照片
2. 在 `开始` 中选择一个照片文件夹
3. 系统递归扫描文件夹，生成缩略图、读取 EXIF、补指纹
4. 后台进入分析管线，输出质量、姿态、物种等结果
5. 系统生成该文件夹的报告与分组结果
6. 用户在 `选片` 中进行筛选、复核、对比、标记
7. 用户导出需要的照片与附加报告
8. 结果沉淀到 `羽迹`

这条线的核心目标是：

- 快速从大量素材里筛出值得保留的照片
- 尽量减少无效翻看与重复判断
- 在模型辅助下，提高筛片效率与准确性

### 3.2 业务线 B：物种沉淀 / 物种墙

这是长期价值模块，不是主筛片流程本身。

核心逻辑为：

1. 每次导入和分析完成后，将识别到的物种结果沉淀到本地中心库
2. 在 `羽迹` 中以照片与物种两种视角浏览长期资产
3. 以物种墙/成就墙的方式展示“我拍到过哪些鸟”
4. 点击物种进入详情页，查看：
   - 当前物种下拍过哪些照片
   - 代表作
   - 时间线
   - 基本介绍
   - 地理分布图
   - 我的拍摄足迹

这条线的核心目标是：

- 把一次次筛片结果沉淀为长期资产
- 提供“成就感 + 知识性 + 可回看性”

## 4. 已确定的顶层命名

最近讨论中已经明确的模块命名如下：

- `开始`
- `选片`
- `羽迹`

说明：

- `开始`：欢迎页 / 初始化入口页
- `选片`：以文件夹为单位的主筛片工作界面
- `羽迹`：长期沉淀模块

`羽迹` 下建议保留两个二级视图：

- `照片`
- `物种`

用户层面不再强调“拍摄批次”概念，统一以“文件夹”为主要组织单位。

## 5. 用户组织单位：文件夹，而不是批次

### 5.1 已确定

前台用户心智统一为：

- 选择一个文件夹开始工作
- 这个文件夹有自己的分析结果、报告、筛片状态、导出记录

不再使用“拍摄批次”作为主要用户概念。

### 5.2 设计原因

原因如下：

- 更符合摄影用户日常整理素材的习惯
- 认知成本更低
- 导入入口更自然
- 文件夹天然可以作为报告统计范围
- “继续上次处理”也更容易理解

### 5.3 内部实现建议

虽然用户看到的是“文件夹”，但系统内部仍应维护一个稳定的文件夹记录对象，用于存储：

- `folder_id`
- 当前路径
- 规范化路径
- 目录身份标识（如果系统可提供）
- 最近扫描时间
- 最近分析时间
- 分析进度
- 导出状态

即：

**用户层以文件夹组织，系统层以中心记录对象维护文件夹生命周期。**

## 6. 存储策略：中心存储为主，显式导出为辅

### 6.1 已确定

已经明确采用以下策略：

- 原始照片仍保留在用户自己的文件夹中
- 分析结果、缩略图、筛片状态、物种记录等集中存储在本地中心库
- 默认不写回原始照片目录
- 如需带走结果，单独提供导出功能

### 6.2 原则

- 用户照片目录只读不写
- 应用数据集中存储在 `~/.plumelens/`
- 中心库是唯一权威存储
- 导出结果是显式动作，不做自动双写

### 6.3 这样做的原因

这套方案更适合支持：

- `选片` 中的继续处理
- `羽迹` 中的跨文件夹汇总
- 模型升级后的重新分析
- 新增种判断
- 文件夹报告版本管理
- 路径失效后的重定位恢复

## 7. 文件夹扫描与增量更新规则

以下规则已经在讨论中明确：

- 默认递归扫描子文件夹
- 同一路径再次打开时，不重新建库，而是继续处理并增量更新
- 先用轻指纹快速建库和可浏览
- 后台继续补完整 hash

增量更新行为建议为：

- 新增文件：建记录并进入分析队列
- 未变化文件：复用已有结果
- 已变化文件：标记需重新分析
- 已删除文件：不立即硬删，标记为 `missing`

## 8. 文件夹改名与重名处理

### 8.1 已处理文件夹改名 / 移动

建议规则：

- 不把文件夹名称当作身份
- 若原路径失效，标记为 `path_missing`
- 用户可通过“重新定位”重新选择新路径
- 优先使用系统目录身份标识恢复
- 其次使用历史照片指纹 / hash 进行内容匹配
- 匹配成功后，恢复原文件夹记录，不丢已有分析结果和筛片状态

### 8.2 文件夹重名

建议规则：

- 重名不是冲突，因为身份不是名字
- 同名但路径不同，视为不同文件夹
- 列表中同时展示：
  - 文件夹名
  - 父路径
  - 最近打开时间
  - 基础统计信息

## 9. 模型管线：目标态为五模型管线

讨论中的目标态已经更新为五模型：

1. `YOLOv26l 新模型`
   鸟类主体检测
2. `CLIPIQA+`
   语义画质评估
3. `HyperIQA`
   技术画质评估
4. `YOLO 姿态/部位模型`
   头、眼、翅膀等关键部位与姿态识别
5. `DINOv3 鸟种识别模型`
   物种分类

## 10. 模型结果的产品含义

产品层面不应只有一个黑盒总分，而应产出两层结果：

### 10.1 筛片结果层

回答“这张值不值得留”：

- 是否有鸟
- 鸟数量
- 语义质量分
- 技术质量分
- 姿态分
- 最终综合分
- 模型评级
- 场景标签
- 问题标签

### 10.2 物种结果层

回答“这是什么鸟、是不是新增种”：

- Top1 物种
- Top-K 候选物种
- 置信度
- 是否新增种

## 11. 评分、分组与排序原则

### 11.1 统一评分，不统一任务语义

两个页面都使用同一套照片级综合评分，但不共享同一套分组语义。

- `选片`：按当前文件夹组织，服务于“做决定”
- `羽迹`：跨文件夹组织，服务于“找内容、回看、沉淀”

### 11.2 选片中的分组与排序

`选片` 不建议直接平刷全部照片，而应采用：

- 一级：文件夹
- 二级：相似组 / 场景组
- 三级：组内按评分排序

相似组或场景组可来自：

- 时间邻近
- 连拍相似
- 视觉相似
- 场景标签

这样更符合真实筛片习惯：

**用户不是在全文件夹里挑一张，而是在一组相似片里挑最好的一张。**

### 11.3 羽迹中的组织方式

`羽迹` 不再按文件夹做强组织，而应支持按以下维度浏览：

- 物种
- 时间
- 地点
- 是否新增种
- 是否已选
- 评分
- 文件夹

## 12. `开始 / 选片 / 羽迹` 的桌面端职责

### 12.1 开始

这是欢迎页与初始化入口页，不是宣传页。

它应负责：

- 解释这是什么软件
- 告诉用户系统是否准备就绪
- 提供主动作：选择照片文件夹
- 提供次动作：继续上次文件夹
- 展示最近文件夹
- 说明从导入到导出的主流程

适用场景：

- 第一次打开
- 当前没有活动文件夹
- 需要从零开始导入新文件夹

### 12.2 选片

这是主工作区，负责：

- 处理一个文件夹内的照片
- 查看文件夹报告
- 按筛选条件和组浏览照片
- 标记 `已选 / 待定 / 淘汰`
- 进行单张复核
- 进入对比模式
- 导出结果

### 12.3 羽迹

这是长期资产模块，负责：

- 跨文件夹浏览照片
- 以物种视角查看长期记录
- 承接物种详情、知识信息与分布图

## 13. `选片` 模块的页面结构

讨论中已形成的稳定结构如下：

### 13.1 主界面布局

- 左侧：文件夹导航
- 中间：当前文件夹工作区
- 右侧：信息抽屉 / 轻复核
- 浮层：深度复核、对比、导出

### 13.2 左栏文件夹导航

建议展示：

- 进行中
- 最近使用
- 路径失效

每个文件夹条目显示：

- 文件夹名
- 父路径
- 状态
- 进度
- 已选数量
- 新增种数量

### 13.3 中间主工作区

从上到下为：

- 文件夹头部
- 文件夹报告
- 筛选与排序工具栏
- 分组后的照片网格

### 13.4 右侧轻复核

用于减少弹出完整复核层的频率。

展示：

- 当前选中照片的较大预览
- 综合分
- 物种
- 问题标签
- EXIF / 技术摘要

### 13.5 深度复核浮层

用于精查单张照片：

- 大图
- 检测框
- 主体裁切
- 眼/头/翅膀 close-up
- 分数拆解
- 候选物种
- 问题原因

### 13.6 对比浮层

用于在相似照片中快速保留最佳一张或少数几张。

## 14. `开始` 页的桌面端职责

`开始` 页应承接：

- 主按钮：选择照片文件夹
- 次按钮：继续上次
- 系统状态：后端、模型、分析链路状态
- 最近文件夹

它的视觉定位应更接近：

**一台本地分析设备的启动页**

而不是营销 landing page。

### 14.1 当前 `开始` 页定稿方向

最近一轮视觉调整后，`开始` 页已经从“多卡片信息堆叠”收敛为更克制的设备启动页：

- 左上为品牌与导航，主体左侧保留一句明确 slogan 与一行辅助说明
- 主动作聚焦为“选择照片文件夹”
- “继续上次文件夹”作为次动作，首次进入或无历史时禁用
- 最近文件夹改为轻量单列模块，只展示少量最近入口
- 无最近文件夹时，该模块自动消失，不制造空卡片
- 系统 / 管线状态降级为底部状态栏，与界面底部齐平
- 右侧使用 Nothing glyph 方向的鸟形灯阵作为品牌视觉，而不是再放功能卡片

已明确去除或降级的内容：

- 无实际交互意义的大型工作流卡片
- 右侧大尺寸管线状态卡片
- 多余的说明句号与过重的文案块
- 为了填空而添加的装饰模块

### 14.2 `开始` 页视觉规格

当前 `开始` 页的视觉约束为：

- 页面不是 landing page，不做营销式 hero
- 标题文案为“阅尽千鸟，鉴翎独选”
- 辅助文案为“智能模型鉴定识别，助您聚焦每一次振翅”
- 中文文案尽量去掉句号，保持短促、硬朗、设备感
- 背景不是纯黑，应保留极低亮度网格与暗部层次
- 最近文件夹是辅助入口，不应压过主动作
- 底部管线状态栏应薄、低调、贴底，不作为卡片悬浮
- 右侧鸟形灯阵应垂直居中于右侧视觉区，不参与左侧布局高度计算
- 灯阵整体排布为圆角矩形面板，内部形成鸟形
- 单个灯点保持点状，基础亮度、常亮点、最高亮点应形成层级
- 光晕要有发光感，但外溢不能过大，避免鸟形变糊

当前灯阵方向可作为后续品牌视觉基准：克制、精密、有光感，但不抢主操作。

## 15. `羽迹` 模块的结构

`羽迹` 建议稳定为两个子视图：

- `照片`
- `物种`

### 15.1 羽迹 - 照片

面向跨文件夹回看：

- 照片网格
- 时间与地点维度
- 评分和筛选
- 物种过滤

### 15.2 羽迹 - 物种

面向长期积累：

- 物种卡片墙
- 是否点亮
- 照片数
- 最佳作品
- 首次拍到 / 最近拍到

### 15.3 物种详情（鸟志）

点击物种后进入详情，建议包含：

- 代表作品
- 全部相关照片
- 基本介绍
- 时间线
- 地理分布图
- 我的拍摄足迹

## 16. 地理分布图方案

最近讨论中明确提出：

物种详情应支持地理分布图。

建议分成两个层级：

1. `物种分布`
   该物种通常出现在什么区域
2. `我的拍摄分布`
   我在哪些地方拍到过它

这样兼顾：

- 科普价值
- 个人成就与回顾价值

## 17. 导出定位

导出属于 `选片` 流程的末端动作，不建议单独拔成一级模块。

导出应支持：

- 导出已选
- 导出已选 + 待定
- 导出当前筛选结果
- 选择目标目录
- 保留原始目录结构
- 附带导出报告 / 清单

## 18. 状态设计原则

### 18.1 模型评级与用户决定分开

这是讨论中反复确认的原则。

模型负责：

- `淘汰 / 记录 / 可用 / 精选`

用户负责：

- `未复核 / 已选 / 待定 / 淘汰`

模型不能替用户最终做决定。

### 18.2 路径丢失不丢结果

文件夹路径失效时：

- 已有分析结果继续保留
- 用户可以重定位恢复
- 不应因为路径变化直接丢掉整个工作上下文

### 18.3 部分分析完成即可开始工作

后台分析不应阻断前台筛片。

只要已有部分结果：

- 用户就应可以开始浏览与初筛

## 19. 桌面端 UX 原则

最近的讨论中，关于产品体验已经形成一些稳定原则：

### 19.1 外壳有风格，工作区让照片说话

视觉层面可以吸收 Nothing 风格，但不能让 UI 抢过照片本身。

### 19.2 先分组，再排序，再复核

桌面端主工作流不应是“平刷全部图后自己翻”，而应：

- 系统先组织
- 用户再筛选
- 最后精查

### 19.3 桌面应用优先

当前交互方案主要面向桌面端大屏工作流：

- 左栏用于切换任务
- 中栏用于看片与决策
- 右栏提供上下文
- 浮层承担复核与对比

### 19.4 不让系统状态压过工作状态

系统状态需要可见，但不应长期抢占主视觉。

典型策略：

- `开始` 页可较强展示系统就绪状态
- `选片` 页中系统反馈应降级为辅助信息

### 19.5 不为兼容极窄窗口牺牲桌面体验

当前产品是桌面端筛片工作台，不需要为极窄窗口做过度适配。

交互与布局应优先保证：

- 正常桌面窗口下的信息密度和稳定性
- 选片工作区可长期使用
- 关键工具栏、照片网格、信息抽屉不因窗口过窄而变形

因此应用应保留合理最小窗口尺寸，当前桌面设计基准不应低于约 `1360 × 860` 的工作区。

当前实现中已经采用较大的最小窗口策略，设计上应继续以桌面大屏为主，不把移动端响应式作为当前阶段目标。

### 19.6 Nothing 风格的使用边界

视觉可以参考 Nothing UI，但不能停留在“黑底 + 细线 + 点阵”的表层。

可吸收的部分：

- 清晰的网格秩序
- 低饱和暗色背景
- 精密的 glyph 灯阵语言
- 克制的红色状态点
- 轻量描边和小半径圆角
- 明确的信息层级

需要避免的部分：

- 为了风格而加入无意义装饰
- 大量卡片悬浮导致页面碎片化
- 所有模块都套同一种玻璃盒
- 标题和状态文字过度加粗
- 光效过度模糊，削弱精密感
- 把系统状态做得比用户任务更醒目

## 20. 当前已确定与建议中的边界

### 20.1 已确定

- 顶层命名：`开始 / 选片 / 羽迹`
- 主组织单位：文件夹
- 扫描方式：递归扫描
- 再次打开同一路径：继续处理并增量更新
- 结果默认中心存储，不写回原始照片目录
- 支持单独导出
- `选片` 与 `羽迹` 分工明确
- `羽迹` 下包含照片与物种视图
- 物种详情支持地理分布图
- `开始` 页不再保留大型工作流预览卡片
- `开始` 页系统状态以底部状态栏表达
- `开始` 页右侧品牌视觉采用鸟形 glyph 灯阵
- 桌面端保留最小窗口尺寸，不兼容极窄窗口

### 20.2 当前建议，但实现中仍可继续打磨

- `选片` 默认按“组”浏览，而不是全平刷
- 新增种建议采用“待确认新增”机制
- `羽迹` 中物种详情的数据源未来可抽象成“资料源”，不要写死 Wiki
- 物种详情中的地图建议区分“物种分布”和“我的拍摄分布”

### 20.3 当前实现中的明确技术债

以下内容已经存在于当前前端实现中，但不应视为最终产品架构：

- `WorkspaceSnapshot` 仍由前端 `useState` 持有
- `createInitialWorkspace` / `createImportedFolder` 仍是 mock 数据生成
- `handleSetDecision`、`handleKeepBestOne` 等业务 mutation 仍在前端直接修改本地状态
- `App.tsx` 仍是大文件，页面与组件尚未按 `pages/`、`components/` 拆分
- `t` 仍通过 props 在多层组件间传递，后续可逐步改为子组件内直接 `useTranslation`

这些技术债在当前阶段可以接受，因为它们服务于快速建立高保真工作流原型。

但进入真实数据阶段时，应优先迁移：

- 文件夹列表与详情：后端 API + TanStack Query
- 导入文件夹：mutation
- 筛片决策：mutation
- 对比保留最佳：mutation
- 导出任务：mutation + 后台任务状态
- 管线状态：以后端 `/health` 与 SSE / task status 为准

### 20.4 当前工程基线

当前已修正并应保持的工程底线：

- `npm run typecheck` 必须真实执行 project references 检查
- renderer TypeScript 目标至少为 `ES2023`，以支持 `toSorted`
- `window.plumelens` 必须由 `env.d.ts` 提供明确类型
- `/health` 前端类型必须包含 `pipeline.ready / pipeline.version / pipeline.models`
- Electron 必须显式启用 `sandbox: true`
- UI 文案应继续通过 i18next 管理，避免新增硬编码可见文案

如果这些底线被破坏，应优先修复，再继续 UI 迭代。
